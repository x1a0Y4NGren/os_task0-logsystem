# 智能日志处理系统

这是一个面向操作系统课程作业的 C 语言项目，主题是“生产者-消费者问题”在智能日志处理场景下的简化实现。
项目运行环境以 Linux / WSL + gcc + pthread 为主，整体风格保持课程作业级别，便于理解线程同步、缓冲区管理和基本文件输出流程。

## 实验背景与任务目标

本实验对应“任务0：智能日志处理系统”。
场景中，生产者线程负责生成日志，消费者线程负责从有界缓冲区中取出日志并进行后续处理。
本项目当前版本的目标是：

- 使用 `pthread_create` 创建多个生产者线程和消费者线程
- 使用互斥锁和条件变量实现有界缓冲区同步
- 让日志真正经过“生产 -> 入缓冲区 -> 出缓冲区 -> 分析 -> 输出文件”的基本流程
- 为后续实验中的参数调整和结果分析保留基础接口

## 当前已实现功能

- 已实现真实的生产者-消费者主流程，不再是单纯的占位线程
- 支持通过命令行参数设置：
  - 缓冲区容量
  - 生产者线程数
  - 消费者线程数
  - 日志总数
- 已实现基于互斥锁和条件变量的循环缓冲区
- 生产者线程会生成示例日志并写入共享缓冲区
- 消费者线程会从共享缓冲区读取日志，调用日志分析模块，再写入结果文件
- 程序运行结束后会生成：
  - `analysis.csv`
  - `alerts.log`

## 项目目录结构

- `src/main.c`：主流程整合，负责参数读取、共享状态初始化、线程创建与回收
- `src/buffer.c`：循环缓冲区实现，负责生产者与消费者之间的同步
- `src/log_ai.c`：日志分类与评分模块的当前实现
- `src/output.c`：分析结果与告警结果输出模块
- `include/common.h`：公共常量与 `LogEntry` 结构体定义
- `include/buffer.h`：缓冲区模块头文件
- `include/log_ai.h`：日志分析模块头文件
- `include/output.h`：输出模块头文件
- `docs/team_plan.md`：当前阶段协作与完成情况说明
- `Makefile`：Linux / WSL 下使用 `gcc -pthread` 的编译规则

## 核心模块说明

### 1. 主流程模块

`main.c` 负责完成以下工作：

- 读取命令行参数
- 初始化共享缓冲区和共享状态
- 创建多个生产者线程和消费者线程
- 统计已生产日志数与已消费日志数
- 在所有生产者结束后写入停止标记，让消费者正常退出
- 释放线程、缓冲区和输出文件资源

### 2. 缓冲区同步模块

`buffer.c` 使用循环队列作为有界缓冲区，核心同步机制为：

- `pthread_mutex_t`：保护缓冲区内部状态
- `pthread_cond_t not_full`：缓冲区满时让生产者等待
- `pthread_cond_t not_empty`：缓冲区空时让消费者等待

当前缓冲区容量不是写死在数组长度里，而是在程序启动时根据参数动态分配。

### 3. 日志分析模块

`log_ai.c` 当前已经不再是“固定写入占位值”的版本，而是一个基于关键词的简化分类与评分模块。
它会根据日志消息中的关键词给出基础分类和分值，例如：

- 命中 `error`、`failed`、`timeout` 等关键词时，可能归为 `ERROR`
- 命中 `warn`、`disk full`、`low memory` 等关键词时，可能归为 `WARNING`
- 命中 `critical`、`panic`、`fatal` 等关键词时，可能归为 `CRITICAL`
- 如果没有命中规则，则归为 `UNCLASSIFIED`

当前评分规则也是简化实现，但已经真实生效：

- 会根据关键词给出基础分数
- 如果消息中带有错误码或较大的数字编号，还会追加分数
- 最后根据分数决定 `needs_alert` 的级别：
  - `0`：无告警
  - `1`：一般告警
  - `2`：较高告警
  - `3`：严重告警

因此，当前版本已经具备“真实分类、真实评分、真实告警判定”的基础能力，只是规则仍然比较朴素，适合作为课程作业初版。

### 4. 输出模块

`output.c` 当前负责生成两个文件：

- `analysis.csv`：保存每条日志的分析结果
- `alerts.log`：保存告警输出

它不再只是初始化文件，还会在程序运行过程中真实写入分析结果和告警内容。
另外，程序结束时还会在终端输出本次运行的简单统计信息，例如处理日志总数、告警总数和各类别数量。

主流程中对输出部分增加了互斥保护，因此在多个消费者线程同时写文件时不会直接发生混乱写入。

## 编译与运行方法

### 编译

在 Linux / WSL 环境下执行：

```bash
make
```

### 运行

```bash
./logsystem
```

也可以带参数运行，例如：

```bash
./logsystem 8 2 1 50
```

表示：

- 缓冲区容量为 `8`
- 生产者线程数为 `2`
- 消费者线程数为 `1`
- 总日志数为 `50`

## 命令行参数说明

程序参数格式如下：

```bash
./logsystem [buffer_capacity] [producer_count] [consumer_count] [total_logs]
```

如果不传参数，将使用默认值：

- `buffer_capacity = 16`
- `producer_count = 2`
- `consumer_count = 1`
- `total_logs = 20`

参数要求：

- `buffer_capacity` 至少为 `1`
- `producer_count` 至少为 `1`
- `consumer_count` 至少为 `1`
- `total_logs` 可以为 `0` 或更大的整数

## 输出文件说明

### 1. `analysis.csv`

该文件保存消费者线程处理后的日志分析结果。
当前表头为：

```text
id,source,category,score,needs_alert,message
```

当前版本中，这个文件会真实写入每条被消费的日志。
例如在 `./logsystem 8 2 1 12` 的一次实际运行中，可以看到：

- `User login request received` 被归为 `UNCLASSIFIED`
- `Disk usage warning detected` 被归为 `WARNING`，分数约为 `45`
- `Service error code reported` 被归为 `ERROR`，分数约为 `75`

### 2. `alerts.log`

该文件用于保存告警输出。
当前文件开头仍会先写入一行初始化文字，但如果日志分数达到告警阈值，就会继续写入真正的告警记录。

例如在 `./logsystem 8 2 1 12` 的一次实际运行中，`alerts.log` 中除了初始化文字外，还会写入由 `Service error code reported` 触发的 `LEVEL-2 [HIGH]` 告警信息。

## 当前项目的特点与限制

### 当前特点

- 已具备真实的线程并发执行流程
- 已实现“生产、缓冲、消费、分析、输出”的完整主链路
- 已支持通过参数调整并发规模和缓冲区容量
- 结构比较朴素，适合操作系统课程作业讲解

### 当前限制

- 日志内容仍是示例数据，不是外部真实日志源
- 分类、评分、告警规则已经实现，但仍是基于关键词的简化规则
- 程序当前没有内置运行时间、吞吐量等统计指标
- 输出文件默认写在项目根目录，而不是单独的结果子目录
- 在多线程情况下，日志写入顺序可能受线程调度影响，不一定严格按日志编号排序
